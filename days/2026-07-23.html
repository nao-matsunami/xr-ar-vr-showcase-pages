今日の重要トピックKey topics today

  1. WebXR Layers API Level 1 は composition layer により、アプリ frame rate と compositor の提示を分け、重要な text / map layer を高解像度に保つ設計を明示している。 WebXR Layers API Level 1 describes composition layers that separate application frame rate from compositor presentation and keep important text or map layers at higher resolution.
  2. `XRProjectionLayer.fixedFoveation` は compositor 側の foveation 量を調整する入口で、周辺視野のコストを下げながら中央の可読性を残す予算設計に使える。 `XRProjectionLayer.fixedFoveation` is an entry point for compositor foveation, useful for reducing peripheral cost while preserving central legibility.
  3. Layers API の depth-sorted-layers は projection / quad / cylinder layer の深度順合成を扱うが、depth 情報が古い layer は正しい順序でも現実との前後関係が破綻しやすい。 The Layers API depth-sorted-layers feature addresses depth ordering among projection, quad, and cylinder layers, but layers with stale depth can still mismatch the real scene.
  4. Depth Sensing Module は `raw` / `smooth` の depth type、CPU / GPU の data usage、`matchDepthView` など、depth の品質と同期を選ぶための判断点が多い。 The Depth Sensing Module exposes several quality and synchronization choices, including `raw` / `smooth` depth types, CPU / GPU data usage, and `matchDepthView`.
  5. Anchors Module は session 内で更新される anchor pose を前提にしているため、anchor drift と layer redraw / depth refresh の優先度を同じ scheduler で見る価値がある。 The Anchors Module assumes anchor poses update within a session, so anchor drift, layer redraw, and depth refresh priority are worth viewing through one scheduler.

表現・インタラクションのアイデアExpression and interaction idea

画面上の gaze point をドラッグすると、中央に近い text / map / tool layer の優先度が上がります。`GPU budget per frame`、`Depth reads per second`、`Head motion`、`Foveation gain` を変えると、緑の更新 layer、黄色の foveated / stale layer、赤の mismatch risk が切り替わります。`Gaze first`、`Depth stale`、`Battery saver` で同じ scene の意思決定が変わるようにしました。

Dragging the gaze point raises the priority of nearby text, map, and tool layers. Changing `GPU budget per frame`, `Depth reads per second`, `Head motion`, and `Foveation gain` switches layers between green updated, amber foveated or stale, and red mismatch risk. `Gaze first`, `Depth stale`, and `Battery saver` show different decisions on the same scene.

制作に使えそうな技術メモProduction note

実装では layer ごとに screen-space area、gaze distance、last depth timestamp、anchor drift、draw cost を持ち、frame ごとに priority queue を作る構成がよさそうです。`fixedFoveation` は単体の見た目設定ではなく、gaze から遠い layer の draw cost を下げる budget knob として扱えます。Depth の `matchDepthView` が成り立たない環境では、depth age を risk に加え、古い depth に頼る layer を早めに再検査対象へ戻す設計が必要です。

A practical implementation can track screen-space area, gaze distance, last depth timestamp, anchor drift, and draw cost per layer, then build a priority queue each frame. `fixedFoveation` is better treated as a budget knob that lowers draw cost for layers far from gaze, not just as a visual setting. When `matchDepthView` is unavailable, depth age should feed into risk so layers relying on old depth return to the refresh queue sooner.

試すならこれIf you try one thing

サンプルで `GPU budget per frame` を 45、`Depth reads per second` を 2、`Head motion` を高めにしてから gaze point を動かしてください。視線近傍だけ緑が残り、遠い layer は foveation によってコストを下げても depth age が伸びると黄色や赤に変わります。`Depth stale` に切り替えると、視線外でも古い depth を持つ layer が優先されます。

Set `GPU budget per frame` to 45, `Depth reads per second` to 2, raise `Head motion`, then move the gaze point. Only near-gaze layers stay green; distant layers get cheaper through foveation but turn amber or red as depth age grows. Switching to `Depth stale` prioritizes old-depth layers even outside gaze.
