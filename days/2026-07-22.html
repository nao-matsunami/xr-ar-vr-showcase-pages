今日の重要トピックKey topics today

  1. WebXR Depth Sensing Module は `depth-sensing` feature descriptor、`usagePreference`、`dataFormatPreference`、`matchDepthView` を通じて、XRSession ごとの depth 取得方針を指定する。 The WebXR Depth Sensing Module uses the `depth-sensing` feature descriptor plus `usagePreference`, `dataFormatPreference`, and `matchDepthView` to describe how an XRSession wants depth data.
  2. Depth 情報は CPU access と WebGL texture access の経路があり、`rawValueToMeters` や `normDepthBufferFromNormView` を使って depth buffer の値を実空間の距離解釈へつなげる。 Depth data can be exposed through CPU access or WebGL texture access, with `rawValueToMeters` and `normDepthBufferFromNormView` connecting depth-buffer values to real-space interpretation.
  3. WebXR Anchors Module は `XRFrame.trackedAnchors` と anchor space の pose 更新を前提にしているため、配置済み object は anchor pose の drift と depth/occlusion の stale 状態を別々に見る必要がある。 The WebXR Anchors Module assumes `XRFrame.trackedAnchors` and updated poses for anchor spaces, so placed objects need separate handling for anchor drift and stale depth/occlusion state.
  4. Hit Test は現時点の real-world surface との交差を返すが、配置後の object すべてに毎 frame hit-test / depth 検査をかけると負荷が膨らむため、距離・サイズ・risk に応じた検査 cadence が重要になる。 Hit Test returns intersections against the current real-world surface estimate, but running hit-test or depth checks for every placed object every frame can be expensive, so cadence based on distance, size, and risk matters.
  5. Immersive Web の proposal samples には AR Depth Sensing CPU/GPU、Plane Detection、Mesh Detection が並び、anchor と depth を組み合わせる時の実装差を追う起点になる。 Immersive Web proposal samples include AR Depth Sensing CPU/GPU, Plane Detection, and Mesh Detection, making them useful references for implementation differences when combining anchors and depth.

注目事例と何が面白いかNotable example and why it matters

今日の注目は Immersive Web の AR Depth Sensing CPU/GPU proposal samples と、Anchors / Hit Test の組み合わせです。Depth Sensing は real world geometry との前後関係を作るために必要ですが、anchor が増えた場面で全 object を同じ頻度で検査すると、見た目に効く object と遠くて小さい object が同じ予算を奪い合います。そこで「近い・大きい・drift が増えた・confidence が低い」anchor へ検査を寄せる設計が面白い点です。

Today's notable reference is the combination of Immersive Web's AR Depth Sensing CPU/GPU proposal samples with Anchors and Hit Test. Depth sensing is needed for real-world front/back relationships, but once many anchors exist, checking every object at the same rate lets small distant objects consume the same budget as visually important ones. The interesting design is to bias checks toward anchors that are near, large, drifting, or low-confidence.

表現・インタラクションのアイデアExpression and interaction idea

画面上に複数の placed anchor card を置き、各 card の depth/occlusion 検査状態を緑、黄、赤で表示します。`Balanced`、`Near first`、`Drift risk`、`Quiet scene` を切り替えると、同じ anchors でも検査対象が変わります。`Burst drift` で tracking が一気に悪化した状態を作り、限られた depth check budget がどの anchor に回るかを観察します。

The screen places multiple anchor cards and marks each card's depth/occlusion check state in green, yellow, or red. Switching `Balanced`, `Near first`, `Drift risk`, and `Quiet scene` changes which anchors get checked. `Burst drift` simulates sudden tracking degradation so the limited depth-check budget visibly moves to the riskiest anchors.

制作に使えそうな技術メモProduction note

実機実装では anchor pose の更新、depth buffer の取得、object bounds の screen-space 投影を分け、scheduler が `次に depth を読む object` を決める形が扱いやすいです。優先度は `1 / distance`、screen area、anchor drift、last check age、tracking confidence、user gaze / pointer proximity を足し合わせます。Depth Sensing の `matchDepthView` が false の場合は古い depth frame の再投影を考える必要があるため、scheduler 側でも depth timestamp の古さを risk に入れると破綻を見つけやすくなります。

On real devices, it is useful to separate anchor-pose updates, depth-buffer acquisition, and screen-space projection of object bounds, then let a scheduler choose which object should read depth next. Priority can combine `1 / distance`, screen area, anchor drift, last-check age, tracking confidence, and user gaze or pointer proximity. If Depth Sensing's `matchDepthView` is false, older depth frames may need reprojection, so adding depth timestamp age to scheduler risk makes failures easier to catch.

今日のミニ試作候補Today's mini prototype candidate

採用した候補は `Anchor Occlusion Scheduler` です。複数 anchor の距離、見かけサイズ、drift、confidence、last check age から risk を計算し、限られた depth check budget を割り当てます。次に足すなら、実際の screen-space bounds と hand / real object mask を重ね、scheduler が選ばなかった object の破綻をユーザー視点で見えるようにします。

The selected prototype is `Anchor Occlusion Scheduler`. It computes risk from distance, apparent size, drift, confidence, and last-check age, then allocates a limited depth-check budget. A useful extension would overlay real screen-space bounds and a hand or real-object mask so the user can see failures on objects the scheduler skipped.
