今日の重要トピックKey topics today

  1. WebXR Depth Sensing Module は 2025-12-10 の Editor's Draft で、XR device が計算した環境の depth 情報を session へ渡す。 The WebXR Depth Sensing Module is an Editor's Draft dated 2025-12-10 and exposes environment depth computed by XR devices to sessions.
  2. Depth Sensing では `depth-sensing` feature descriptor と `depthSensing` 初期化設定が重要で、CPU / GPU usage と depth data format を選ぶ必要がある。 Depth Sensing depends on the `depth-sensing` feature descriptor plus `depthSensing` session initialization, including CPU/GPU usage and depth data format choices.
  3. `XRDepthInformation.rawValueToMeters` は raw depth value を meter に変換する scale で、今回の自作サンプルでは疑似 depth map を直接 meter として扱った。 `XRDepthInformation.rawValueToMeters` scales raw depth values into meters; today's sample treats its simulated depth map directly as meter values.
  4. Lighting Estimation は `XRLightProbe` と `XRLightEstimate` を通じて primary light と spherical harmonics を扱い、遮蔽後の見え方の次段階になる。 Lighting Estimation uses `XRLightProbe` and `XRLightEstimate` for primary light and spherical harmonics, making it a natural next step after occlusion.
  5. Hand Input は 25 joints の articulated hand pose を扱うため、depth で隠れた仮想物に手が近づく表現へ発展させやすい。 Hand Input exposes articulated hand poses across 25 joints, which can extend this depth idea into hands approaching partially hidden virtual objects.

注目事例と何が面白いかNotable example and why it matters

今日の注目は WebXR Depth Sensing Module です。面白い点は、AR で『そこに置けた』だけでは足りず、現実の机や人の輪郭より奥にある仮想物を正しく隠すことで初めて合成の説得力が上がることです。仕様上は raw / smooth、CPU / GPU、複数 data format の選択があり、体験上は depth の精度よりも遮蔽の破綻をどう見せるかが重要になります。

Today's notable reference is the WebXR Depth Sensing Module. What matters is that AR placement alone is not enough: compositing becomes convincing when virtual content behind real desks or silhouettes is hidden correctly. The specification exposes choices such as raw/smooth depth, CPU/GPU usage, and data formats, while the experiential issue is how to present imperfect occlusion.

表現・インタラクションのアイデアExpression and interaction idea

depth map を単なる背景処理にせず、ユーザーが置いたカードの何割が現実側に隠れているかを meter として出すと、AR の不確かな合成状態を説明できます。今日のサンプルでは、ドラッグでカード位置を変え、深度を手前・奥へ動かし、Mask を切り替えて遮蔽の有無を比べられるようにしました。

Instead of treating the depth map as invisible background processing, showing how much of a placed card is hidden by the real scene makes AR's uncertain compositing state legible. Today's sample lets the user drag the card, move its depth near or far, and toggle the occlusion mask.

制作に使えそうな技術メモProduction note

実 WebXR では `navigator.xr.requestSession("immersive-ar", { requiredFeatures: ["depth-sensing"], depthSensing: { usagePreference, dataFormatPreference } })` のように session を作り、frame / view ごとの depth information を取得します。CPU 側で判定するなら `getDepthInformation(view)` と `rawValueToMeters`、GPU 側なら depth texture と shader で virtual fragment depth を比較する構成になります。

In real WebXR, a session can request depth sensing with options such as `requiredFeatures: ["depth-sensing"]` and a `depthSensing` configuration for usage and data format preferences. CPU-side logic would use per-view depth information and `rawValueToMeters`; GPU-side compositing would compare virtual fragment depth against a depth texture in a shader.
