今日の重要トピックKey topics today

  1. WebXR Layers API Level 1 の 2026-06-12 版 Editor's Draft は composition layer の利点として、compositor frame rate での提示、再投影、低遅延、可読性、電力効率を整理している。 The WebXR Layers API Level 1 Editor's Draft dated 2026-06-12 frames composition layers around compositor-rate presentation, reprojection, lower latency, legibility, and power efficiency.
  2. 同仕様では XRProjectionLayer が全視野を埋める layer とされ、fixedFoveation、ignoreDepthValues、deltaPose などが depth と補間の設計点になる。 The same specification defines XRProjectionLayer as a full-view layer, with fixedFoveation, ignoreDepthValues, and deltaPose becoming important design points for depth and synthesis.
  3. Depth sorting between layers は projection layer、quad/cylinder/media layer、前景 input 表現を同時に扱うときの破綻点になりやすく、どの layer の depth を compositor が信用するかが重要になる。 Depth sorting between layers is a likely failure point when projection layers, quad/cylinder/media layers, and foreground input visualization are composed together.
  4. Space Warp では projection layer に color だけでなく motion vector や depth が絡むため、前景 pointer や hand mesh が別タイミングで更新されると、背景だけが補正されて相対位置がずれる可能性がある。 Space Warp can involve color, motion vectors, and depth for projection layers, so foreground pointers or hand meshes updated on another schedule can drift relative to a compensated background.
  5. Immersive Web の Input Selection Space Warp sample は、入力イベントと projection layer、Space Warp、motion vector、depth を組み合わせる実装例として、今日の前景レイヤー観察のよい参照になる。 The Immersive Web Input Selection Space Warp sample is a useful implementation reference because it combines input events with a projection layer, Space Warp, motion vectors, and depth.

表現・インタラクションのアイデアExpression and interaction idea

背景の projection layer は cyan の最後に app が描いた pose と green の補正 pose で表示し、前景の pointer/hand layer は rose で毎フレーム現在 pose に合わせます。ユーザーがドラッグすると前景 ray が動き、Render rate や Warp gain を下げると green の補正位置と white の現在 pose のズレが広がります。空間 UI の接地点が『見た目は当たっているのに depth 判定はずれる』状態を短時間で確認できます。

The background projection layer is shown as cyan for the last app-rendered pose and green for the compensated pose, while the foreground pointer/hand layer is drawn in rose at the current pose every frame. Dragging moves the foreground ray, and lowering render rate or warp gain increases the gap between the green compensated position and the white current pose. This makes it quick to inspect cases where a spatial UI contact point appears aligned but depth judgment disagrees.

制作に使えそうな技術メモProduction note

実装では `XRProjectionLayer.deltaPose`、`ignoreDepthValues`、Space Warp 用の `motionVectorTexture` / `depthStencilTexture` の有無、そして `XRSession.updateRenderState({ layers })` の layer 順序を一緒に見る必要があります。前景 pointer を projection layer 内に描くのか、別の quad/cylinder layer や通常 scene geometry として扱うのかで、補間・depth sorting・hit test の基準が変わります。今日のサンプルはその判断を HMD なしで試すため、panel depth と hand depth をスライダー化しました。

A real implementation should consider `XRProjectionLayer.deltaPose`, `ignoreDepthValues`, the availability of Space Warp `motionVectorTexture` / `depthStencilTexture`, and layer ordering via `XRSession.updateRenderState({ layers })` together. The interpolation, depth sorting, and hit-test basis changes depending on whether the foreground pointer is drawn inside the projection layer, as another quad/cylinder layer, or as ordinary scene geometry. Today's sample exposes panel depth and hand depth as sliders to explore that decision without a headset.

試すならこれIf you try one thing

Open Foreground Warp Mismatch Scope, lower Render rate near 8 Hz and Warp gain near 0.4, then drag the canvas left and right. Screen mismatch increases and the rose foreground pointer separates from the green compensated panel contact point. Moving Hand depth behind Panel depth switches the depth decision to `panel first`, making ordering problems visible too.
