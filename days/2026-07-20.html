今日の重要トピックKey topics today

  1. WebXR Hit Test Module の最新 Editor's Draft は 2025-12-11 で、`hit-test` feature descriptor、`XRHitTestSource`、`XRTransientInputHitTestSource`、`XRFrame.getHitTestResults()` / `getHitTestResultsForTransientInput()` を中心に現実空間との交差結果を扱う。 The latest WebXR Hit Test Module Editor's Draft is dated 2025-12-11 and centers on the `hit-test` feature descriptor, `XRHitTestSource`, `XRTransientInputHitTestSource`, and `XRFrame.getHitTestResults()` / `getHitTestResultsForTransientInput()`.
  2. Hit Test はアプリ側の仮想シーンへの raycast ではなく、AR デバイスが理解している現実世界 geometry への問い合わせとして定義されているため、reticle の信頼度は surface 側の更新品質にも依存する。 Hit Test is defined as querying real-world geometry understood by the AR device, not raycasting against the application's virtual scene, so reticle confidence depends on surface update quality.
  3. WebXR Hand Input Module は `XRInputSource.hand` と 25 joint の `XRJointSpace` を定義しており、隠れた joint の扱いは emulation または手全体の null pose になり得るため、選択 UI には hold / hysteresis が必要になる。 The WebXR Hand Input Module defines `XRInputSource.hand` and 25 `XRJointSpace` joints; obscured joints can be emulated or represented by null poses for the whole hand, so selection UI needs hold and hysteresis behavior.
  4. WebXR DOM Overlays は immersive session 中に単一 DOM element を透明背景の 2D rectangle として表示する仕組みで、AR reticle や hand UI の debug / control layer と組み合わせやすい。 WebXR DOM Overlays can show a single transparent 2D DOM rectangle during an immersive session, making it useful as a debug or control layer around AR reticles and hand UI.
  5. Immersive Web の公式 samples は Hit Test、Hit Test with Anchors、Hand Tracking、Input Selection を分けて示しており、今日の試作ではそれらの入力経路を HMD 不要の最小モデルとして接続した。 Immersive Web's official samples separate Hit Test, Hit Test with Anchors, Hand Tracking, and Input Selection; today's prototype connects those input paths as a headset-free minimal model.

注目事例と何が面白いかNotable example and why it matters

今日の注目は WebXR Hit Test Module と Immersive Web の Hit Test / Hand Tracking / Input Selection samples の組み合わせです。Hit Test は reticle を現実面へ吸着させる入口で、Hand Input は 25 joint の姿勢から選択意図を読む入口です。面の confidence と手の confidence を別々に扱うと、surface は見えているが手が荒れている、手は安定しているが hit-test 面が弱い、という失敗状態を UI に反映できます。

Today's notable combination is the WebXR Hit Test Module plus the Immersive Web Hit Test, Hand Tracking, and Input Selection samples. Hit Test is the path for attaching a reticle to a real-world surface, while Hand Input is the path for interpreting selection intent from 25 hand joints. Keeping surface confidence and hand confidence separate lets the UI represent failure states such as a good surface with noisy hands or stable hands with a weak hit-test plane.

表現・インタラクションのアイデアExpression and interaction idea

手先から surface へ伸びる ray、surface 上の reticle、joint / hit / stability / risk の 4 本メーターを同時に表示します。`Steady`、`Jitter`、`Weak plane`、`False tap` の preset と slider で、reticle が lock する条件と hold へ落ちる条件を観察できます。表現としては、reticle を常に描くのではなく、条件が揃った時だけ太く緑にして、危険時は search / hold の見た目へ戻します。

The sample displays a ray from the hand tip to the surface, a reticle on the surface, and four meters for joint, hit, stability, and risk. Presets and sliders for `Steady`, `Jitter`, `Weak plane`, and `False tap` make it easy to observe when the reticle locks or falls back to hold. The visual approach avoids treating the reticle as always valid: it becomes thick and green only when evidence is stable, otherwise it returns to search or hold.

制作に使えそうな技術メモProduction note

実機実装では `requestHitTestSource()` や `requestHitTestSourceForTransientInput()` の結果を frame ごとに読む経路と、`inputSource.hand` / `XRFrame.getJointPose()` の経路を同じ state machine に入れると扱いやすいです。`score > threshold` だけで即配置せず、数 frame の lock age と hysteresis を持たせると、手の一瞬の欠損や面推定の揺れで false select が起きにくくなります。

On a real device, feeding both the per-frame hit-test path from `requestHitTestSource()` or `requestHitTestSourceForTransientInput()` and the hand path from `inputSource.hand` / `XRFrame.getJointPose()` into the same state machine should be manageable. Instead of placing immediately when `score > threshold`, keeping a lock age and hysteresis over several frames helps avoid false selects caused by brief joint dropout or surface-estimation jitter.

試すならこれIf you try one thing

`Steady` で reticle が緑に lock することを確認し、次に `False tap` を押してください。joint confidence と hit-test confidence が残っていても、jitter と depth mismatch が大きいと false select risk が上がり、decision が `HOLD SELECT` へ落ちます。最後に Hysteresis を下げると復帰は速くなりますが、揺れへの耐性が落ちることが分かります。

Start with `Steady` and confirm that the reticle locks in green, then press `False tap`. Even when some joint and hit-test confidence remain, high jitter and depth mismatch increase false-select risk and push the decision to `HOLD SELECT`. Lowering Hysteresis makes recovery faster but reduces resistance to noisy input.

今日のミニ試作候補Today's mini prototype candidate

採用した候補は `Hand Reticle Stability Gate` です。手先 ray、surface confidence、depth mismatch、hysteresis をまとめて、`LOCKED RETICLE`、`SEARCHING SURFACE`、`HOLD SELECT` の 3 状態へ落とし込みます。次に足すなら、lock 後に anchor candidate を複数保存し、drift が一定以上になった時だけ再 hit-test する比較表示です。

The selected prototype is `Hand Reticle Stability Gate`. It combines hand-tip ray, surface confidence, depth mismatch, and hysteresis into three states: `LOCKED RETICLE`, `SEARCHING SURFACE`, and `HOLD SELECT`. A useful extension would save multiple anchor candidates after lock and compare re-hit-testing only when drift passes a threshold.
