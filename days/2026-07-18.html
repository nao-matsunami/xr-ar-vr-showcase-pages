今日の重要トピックKey topics today

  1. WebXR Device API の最新公開版は 2026-03-16 の Candidate Recommendation Draft で、VR/AR デバイスへの入力・出力、session、frame loop、pose、input source の基礎を定義している。 The latest published WebXR Device API is the Candidate Recommendation Draft dated 2026-03-16, defining the basics of input, output, sessions, frame loops, poses, and input sources for VR/AR devices.
  2. WebXR Layers API Level 1 の 2026-06-12 Editor's Draft は、composition layer の利点として compositor frame rate、再投影、可読性、電力効率、低遅延を整理している。 The WebXR Layers API Level 1 Editor's Draft dated 2026-06-12 frames composition layers around compositor frame rate, reprojection, legibility, power efficiency, and low latency.
  3. 同 Layers API では XRProjectionLayer が全視野 layer として定義され、fixedFoveation、deltaPose、depthStencilTexture、motionVectorTexture が描画品質と補間の判断点になる。 In the Layers API, XRProjectionLayer is the full-view layer, with fixedFoveation, deltaPose, depthStencilTexture, and motionVectorTexture becoming important quality and synthesis decisions.
  4. Immersive Web の WebGPU samples には XRGPUBinding と XRProjectionLayer を使う immersive VR / AR の例が並び、WebGL とは別の描画経路を試す入口になっている。 The Immersive Web WebGPU samples include immersive VR / AR examples using XRGPUBinding and XRProjectionLayer, making them a practical entry point for the WebGPU rendering path.
  5. ImmersiveWeb.dev の対応表では WebXR/WebGPU Bindings は未対応扱いのままで、現時点では実装状況の確認 UI と graceful fallback が重要になる。 ImmersiveWeb.dev still marks WebXR/WebGPU Bindings as unsupported, so capability checks and graceful fallback remain central design work.

注目事例と何が面白いかNotable example and why it matters

今日の注目は Immersive Web の WebGPU samples です。WebXR の immersive VR / AR session を WebGPU 経由で描画し、XRGPUBinding と XRProjectionLayer を使う例が用意されています。前日まで見てきた Space Warp、depth、foreground layer の話は WebGL 前提でも考えられますが、WebGPU 経路では texture format、depth attachment、描画負荷、対応ブラウザの判定をより明示的に扱う必要があります。

Today's notable reference is the Immersive Web WebGPU samples. They include immersive VR / AR sessions rendered through WebGPU using XRGPUBinding and XRProjectionLayer. The Space Warp, depth, and foreground-layer questions from previous days can be reasoned about in WebGL too, but the WebGPU path makes texture format, depth attachments, rendering pressure, and browser capability checks more explicit.

表現・インタラクションのアイデアExpression and interaction idea

ブラウザ、XRSession、XRGPUBinding、Projection、Compositor を横並びの pipeline として描き、各段階の readiness を色とバーで見せます。ユーザーは gaze point をドラッグし、texture format、foveation、depth bits、layer count、eye scale を変えることで、WebGPU 経路でどこに圧力が寄るかを観察できます。WebGPU が未検出でも UI は動き、fallback として設計判断を試せるようにしました。

The sample draws Browser, XRSession, XRGPUBinding, Projection, and Compositor as a horizontal pipeline, using color and bars to show readiness at each stage. Users can drag the gaze point and change texture format, foveation, depth bits, layer count, and eye scale to inspect where pressure moves in a WebGPU path. Even when WebGPU is not detected, the UI still runs as a fallback design probe.

制作に使えそうな技術メモProduction note

実装時は `navigator.gpu`、`navigator.xr`、`XRGPUBinding` の有無を別々に見ます。WebGPU があるだけでは WebXR に WebGPU texture を渡せるとは限らず、XRGPUBinding と XRProjectionLayer を使う path が必要です。texture format は `bgra8unorm` / `rgba8unorm` / `rgba16float` の議論があり、HDR や copy cost、runtime の native support を見ながら選ぶことになります。

A practical implementation should check `navigator.gpu`, `navigator.xr`, and `XRGPUBinding` separately. Having WebGPU does not necessarily mean WebGPU textures can be submitted to WebXR; the XRGPUBinding and XRProjectionLayer path is needed. Texture format discussions around `bgra8unorm`, `rgba8unorm`, and `rgba16float` imply that HDR, copy cost, and native runtime support must be considered together.

試すならこれIf you try one thing

`XRGPUBinding Readiness Matrix` を開き、Stress を押してから Format を `rgba16float`、Layer count を 5、Eye scale を 1.3 付近にしてください。Texture pressure と frame budget が増え、WebGPU 経路で HDR layer や高解像度 projection layer を入れると compositor までの余裕が減ることを確認できます。Mobile preset では foveation を強めた設計の見え方を試せます。

Open XRGPUBinding Readiness Matrix, press Stress, set Format to `rgba16float`, Layer count to 5, and Eye scale near 1.3. Texture pressure and frame budget rise, showing how HDR layers or high-resolution projection layers can reduce compositor headroom in a WebGPU path. The Mobile preset lets you inspect a stronger foveation design.

俯瞰メモBird's-eye memo

これまでのサンプルは、奥行き HUD、hit test / anchor、depth occlusion、lighting、hand input、Layers、360 media、redraw、Space Warp、foreground mismatch と進み、今日 WebGPU binding の入口に来ました。体験側の問題は相変わらず『何をどの layer で、どの depth と更新頻度で渡すか』ですが、WebGPU 経路では format と attachment の選択がそのまま品質・遅延・電力に跳ね返ります。

The sequence has moved through depth HUD, hit test and anchors, depth occlusion, lighting, hand input, Layers, 360 media, redraw, Space Warp, foreground mismatch, and now the WebGPU binding entry point. The experience-level problem is still what to submit in which layer, with which depth and update cadence, but the WebGPU path makes format and attachment choices directly affect quality, latency, and power.

今日のミニ試作候補Today's mini prototype candidate

採用した候補は `XRGPUBinding Readiness Matrix` です。実ブラウザの `navigator.gpu` / `navigator.xr` / `XRGPUBinding` を確認しつつ、WebGPU がない環境でも texture pressure、frame budget、readiness を操作できるようにしました。次に発展させるなら、WebGPU compute pass 風の joint confidence filter を足し、手入力と projection layer の depth / format 設計を同じ画面で比べます。

The selected prototype is XRGPUBinding Readiness Matrix. It checks real `navigator.gpu`, `navigator.xr`, and `XRGPUBinding` availability while still allowing texture pressure, frame budget, and readiness to be manipulated without WebGPU support. A natural extension is adding a WebGPU compute-pass-style joint confidence filter and comparing hand input with projection-layer depth / format design in the same screen.
