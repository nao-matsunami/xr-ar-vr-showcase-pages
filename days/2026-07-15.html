今日の重要トピックKey topics today

  1. WebXR Layers API Level 1 は 2026-06-12 版 Editor's Draft で、composition layer の性能、可読性、電力、遅延面の利点を整理している。 The WebXR Layers API Level 1 Editor's Draft dated 2026-06-12 describes composition layer benefits around performance, legibility, power, and latency.
  2. XRCompositionLayer.needsRedraw は、layer が次の XR animation frame で再描画されるべきことを示し、再描画しないと flicker などの副作用があり得る。 XRCompositionLayer.needsRedraw indicates that a layer should be rerendered in the next XR animation frame, otherwise flickering or other side effects may occur.
  3. composition layer の underlying resources が失われた場合、needsRedraw が true になり、projection layer 以外では redraw event が queue される。 When a composition layer's underlying resources are lost, needsRedraw becomes true and non-projection layers queue a redraw event.
  4. XRWebGLBinding.createQuadLayer の isStatic は、needsRedraw が true のときだけ描ける layer を表し、静的注釈や地図のような高解像度 layer に向く。 The isStatic option on XRWebGLBinding.createQuadLayer represents a layer that can only be drawn when needsRedraw is true, which suits static annotations or maps.
  5. XRProjectionLayer.fixedFoveation は中心以外を低解像度にして GPU 負荷を下げる仕組みで、背景向きだが文字や細部には慎重な調整が必要。 XRProjectionLayer.fixedFoveation reduces GPU load by lowering edge resolution outside the center, making it useful for backgrounds but risky for text or fine detail.

注目事例と何が面白いかNotable example and why it matters

今日の注目は WebXR Layers API の static layer と redraw まわりです。面白い点は、空間内の文字、地図、注釈のような高解像度で読みたい layer を毎フレーム再描画しなくても、compositor が pose に合わせて再投影できる設計になっていることです。映像やポインタは動かし続けつつ、静的情報は needsRedraw のタイミングだけ更新する、という分業が見えてきます。

Today's notable area is static layers and redraw behavior in the WebXR Layers API. The interesting part is that high-resolution readable layers such as text, maps, or annotations do not necessarily need to be redrawn every frame because the compositor can reproject them with pose updates. Video and pointers can keep moving while static information updates only when needsRedraw requires it.

表現・インタラクションのアイデアExpression and interaction idea

静的な説明パネルは isStatic な quad layer として扱い、視線カーソルや手先の ray は小さな動的 layer として高頻度に更新する構成が有効です。今日のサンプルでは Static と Dynamic を切り替え、Static では注釈 layer の再描画回数がほぼ増えず、ポインタだけが動き続ける様子を counters で見られます。Loss ボタンは layer resource loss 後の redraw event を擬似的に表します。

A useful composition is to treat explanatory panels as static quad layers, while gaze cursors or fingertip rays update as small dynamic layers at a higher frequency. Today's sample switches between Static and Dynamic modes: in Static mode the annotation layer's redraw count barely changes while the pointer keeps moving. The Loss button simulates a redraw event after layer resource loss.

制作に使えそうな技術メモProduction note

実装時は `navigator.xr.requestSession("immersive-ar", { optionalFeatures: ["layers"] })` で layers を要求し、`XRWebGLBinding.createQuadLayer({ space, viewPixelWidth, viewPixelHeight, isStatic: true })` のように静的 quad を作る流れを検討します。`XRSession.updateRenderState({ layers: [...] })` の配列は back-to-front 順で扱われるため、背景 projection、静的注釈、動的ポインタの順序を明示します。projection layer の `fixedFoveation` は背景の負荷削減には使えますが、文字 layer に強くかける設計は避けたいです。

For a real implementation, request layers through `navigator.xr.requestSession("immersive-ar", { optionalFeatures: ["layers"] })`, then create a static quad with a shape such as `XRWebGLBinding.createQuadLayer({ space, viewPixelWidth, viewPixelHeight, isStatic: true })`. The `XRSession.updateRenderState({ layers: [...] })` array is treated back-to-front, so the projection background, static annotations, and dynamic pointer layers should be ordered explicitly. `fixedFoveation` can reduce background load, but it should be applied carefully around text layers.
