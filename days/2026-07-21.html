今日の重要トピックKey topics today

  1. WebXR Anchors Module の Editor's Draft は 2026-04-27 で、`anchors` feature descriptor、`XRAnchor`、`XRFrame.trackedAnchors`、anchor pose の更新と削除を中心に、基盤システムに特定 pose の追跡を依頼する仕組みを定義している。 The WebXR Anchors Module Editor's Draft is dated 2026-04-27 and defines the `anchors` feature descriptor, `XRAnchor`, `XRFrame.trackedAnchors`, and anchor pose update/removal concepts for asking the underlying system to track a specific pose.
  2. Anchors は hit-test の瞬間的な交差点を長く使うための橋渡しになるが、デバイス側の world understanding が変わると anchor pose も更新されるため、配置 UI には drift の観測と再固定の方針が必要になる。 Anchors bridge a momentary hit-test intersection into longer-lived placement, but the anchor pose can update as the device's world understanding changes, so placement UI needs a policy for observing drift and re-locking.
  3. WebXR Hit Test Module は plane / point / mesh などの trackable type と `XRHitTestSource` を扱うため、anchor 作成前の新鮮な surface estimate と anchor 作成後の tracked pose を分けて考えやすい。 The WebXR Hit Test Module covers trackable types such as plane, point, and mesh plus `XRHitTestSource`, which makes it useful to separate fresh surface estimates before anchor creation from tracked poses after anchor creation.
  4. WebXR Device API の 2026-03-16 Candidate Recommendation Draft は frame loop、reference space、pose、feature descriptor の土台で、anchor / hit-test / DOM overlay などの module はこのセッションモデルに乗る。 The 2026-03-16 WebXR Device API Candidate Recommendation Draft provides the frame loop, reference space, pose, and feature descriptor foundation that modules such as anchors, hit test, and DOM overlay build on.
  5. Immersive Web の公式 samples には Anchors、Hit Test、Hit Test with Anchors が分かれており、今日の試作では公開カードの中心を外部デモではなく、自作の drift / re-hit 判断モデルにした。 Immersive Web's official samples split Anchors, Hit Test, and Hit Test with Anchors; today's prototype keeps the public card centered on an original drift / re-hit decision model rather than external demos.

注目事例と何が面白いかNotable example and why it matters

今日の注目は Immersive Web の Hit Test with Anchors sample と WebXR Anchors Module の組み合わせです。Hit Test は「今見えている面」への交差を返し、Anchors は「この pose を追跡し続けたい」という依頼に変わります。面推定が更新されるほど anchor の pose も微妙に動くため、置いた後に drift をどう UI 化するかが実アプリの質に直結します。

Today's notable combination is the Immersive Web Hit Test with Anchors sample and the WebXR Anchors Module. Hit Test returns an intersection with the currently understood surface, while Anchors turn that into a request to keep tracking a pose. As surface estimation updates, anchor poses can move subtly, so the way an app surfaces drift after placement directly affects quality.

表現・インタラクションのアイデアExpression and interaction idea

fresh hit-test の reticle、保存した anchor pose、現在追跡されている live pose を 1 画面に分けて描きます。anchor から live pose へ点線を伸ばし、drift がしきい値を超えると黄色の警告表示に変えます。`Save anchor`、`Re-hit worst`、`Auto re-hit` を用意し、置いた object を無条件に信用するのではなく、必要な時だけ surface を取り直す流れを見せます。

The sample draws the fresh hit-test reticle, saved anchor pose, and currently tracked live pose separately. A dashed line connects the anchor to its live pose, and anchors turn into a yellow warning state when drift passes the threshold. With `Save anchor`, `Re-hit worst`, and `Auto re-hit`, the interaction shows a flow that refreshes the surface only when needed instead of blindly trusting placed objects.

制作に使えそうな技術メモProduction note

実機では `requestHitTestSource()` の結果から anchor を作成し、frame ごとに `XRFrame.trackedAnchors` と `XRFrame.getPose(anchor.anchorSpace, referenceSpace)` を読む構成が基本になります。drift は過去 pose と現在 pose の距離だけでなく、surface confidence、tracking loss、user motion、object の重要度で重み付けすると扱いやすいです。再 hit-test は毎 frame ではなく、一定距離、一定時間、またはユーザー注視時に限定すると UX と負荷のバランスが取りやすくなります。

On a real device, the basic path is to create anchors from `requestHitTestSource()` results, then read `XRFrame.trackedAnchors` and `XRFrame.getPose(anchor.anchorSpace, referenceSpace)` each frame. Drift works better when weighted by surface confidence, tracking loss, user motion, and object importance rather than just previous-versus-current pose distance. Re-hit testing should be limited by distance, time, or user focus instead of running every frame.

試すならこれIf you try one thing

`Stable table` で anchor を保存し、`Walk around` に切り替えて drift が増える様子を見てください。`Auto re-hit on` のままだと stale anchor が周期的に再固定され、off にすると黄色い queue が溜まります。`Re-hit threshold` を下げると安全寄り、上げると anchor を信用する寄りの挙動になります。

Save an anchor in `Stable table`, then switch to `Walk around` and watch drift rise. With `Auto re-hit on`, stale anchors are periodically re-locked; with it off, the yellow queue accumulates. Lowering `Re-hit threshold` makes the policy more conservative, while raising it trusts anchors longer.

The prior samples covered input confidence, hit-test reticles, and hand stability. Today's sample moves from the moment of placement to what happens after placement, addressing the instability of AR objects updating along with real-world surface understanding. XR placement UX must design not only the success moment but also when to repair objects that slowly drift afterward.

今日のミニ試作候補Today's mini prototype candidate

採用した候補は `Anchor Drift Re-hit Lab` です。複数 anchor を保存し、surface confidence、tracking noise、world drift、anchor memory、re-hit threshold を変えながら、stale anchor だけを queue に入れます。次に足すなら、anchor の近さと object サイズに応じて depth occlusion の検査頻度を変える scheduler です。

The selected prototype is `Anchor Drift Re-hit Lab`. It saves multiple anchors and queues only stale anchors while varying surface confidence, tracking noise, world drift, anchor memory, and re-hit threshold. A useful extension would be a scheduler that changes depth-occlusion check frequency based on anchor proximity and object size.
